# v18 Contracts: Browser Evidence and Leasing Design

## Objective
Define the operational contracts for how APR requests and consumes browser-based interactions via Oracle. This set of schemas provides the boundaries between APR's orchestrator and the brittle DOM-automation layer handled by Oracle.

## Core Concepts

### Browser Lease
The `browser-lease.schema.json` defines the agreement between APR and the provider slot. It issues a time-bound lease (`ttl_seconds`, `issued_at`, `expires_at`) indicating that a specific provider (e.g., `chatgpt`, `gemini`) has an available browser automation resource. It maps the `profile_id_hash` to ensure the correct user profile/cookies are used without exposing them.

### Browser Session
The `browser-session.schema.json` represents an active remote browser connection. Once a lease is acquired, it becomes a session (`session_id`) that locks the provider (`provider_locks`).

### Browser Evidence
The `browser-evidence.schema.json` is the most critical compliance contract. It proves that the Oracle automation successfully navigated the DOM, verified the mode (`mode_verified`), selected the correct reasoning effort (`reasoning_effort_verified`), and correctly parameterized the model *before* the prompt was submitted.
To preserve privacy and security, this contract **explicitly forbids**:
- `cookies`
- `raw_dom`
- `screenshots` (unless explicitly authorized)
- `private_prompt_body`
- `browser_auth_material`
Any JSON artifact containing these keys will fail schema validation.

### Remote Browser Endpoint
The `remote-browser-endpoint.schema.json` tracks available host environments (`host_env`, `host_hash`) and validates that no plaintext secrets are leaked (`no_plaintext_secrets`). It governs the endpoints Oracle will connect to for automation.

## Usage
These schemas act as the single source of truth for the artifacts generated by Oracle and parsed by APR. Any E2E tests validating provider routes should assert against these schemas.
